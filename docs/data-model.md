# CRM для обработки лидов — модель данных

## О документе
Фиксируются сущности, атрибуты, связи и бизнес-ограничения для CRM обработки лидов. Типы колонок описаны в терминах **PostgreSQL**; перечисления задаются как отдельные **`ENUM`**. DDL таблиц (`CREATE TABLE`, индексы) выносится в SQL-артефакты проекта.

## Список сущностей

| Сущность | Назначение |
|----------|------------|
| `Lead` | Объект лида на всём жизненном цикле до/после конвертации. |
| `Client` | Клиент, созданный после успешной конвертации лида. |
| `Manager` | Ответственный менеджер (справочник пользователей процесса). |

## Концептуальные связи

```text
Manager 1 ──< Lead (назначение: assignee_id)
Lead      1 ── 0..1 Client (конвертация: Client.lead_id уникален)
Lead      ── 0..1 Lead (повторное обращение: reopen_of_lead_id → предыдущий лид)
```

## Перечисления PostgreSQL (`ENUM`)

В PostgreSQL перечисление — это не «поле просто текстом», а **пользовательский тип**, объявленный через `CREATE TYPE имя AS ENUM (...)`. В `CREATE TABLE` у колонки указывается **имя этого типа** (здесь: `lead_source`, `lead_status`), а не слово `ENUM`.

В таблицах атрибутов ниже колонка **ENUM** содержит **полный допустимый набор меток**. Колонку **Тип PostgreSQL** для таких полей обозначаем **`ENUM`** (класс типа в этой модели), в едином стиле с `BIGINT`, `TEXT`, `TIMESTAMPTZ`, `BOOLEAN` — фактическое имя типа в DDL указано рядом в описании поля.

```sql
CREATE TYPE lead_status AS ENUM (
  'Новый',
  'В работе',
  'Отложен',
  'Успешно',
  'Ложный',
  'Отказ'
);

CREATE TYPE lead_source AS ENUM (
  'web',
  'phone',
  'chat',
  'social',
  'manual'
);
```

Явного `ENUM` для причин закрытия нет: причины хранятся как свободный текст в `close_reason`.

## Кардинальность и роли связей

- **`Manager` ↔ `Lead`:** один менеджер может быть назначен на множество лидов через `Lead.assignee_id`.
- **`Lead` ↔ `Client`:** каждая успешная конвертация даёт одного клиента для одного лида; лиды без конвертации не имеют `Client`.
- **`Lead` ↔ `Lead` (`reopen_of_lead_id`):** не более одного «родительского» лида на повторную заявку; цепочка повторных обращений не раздувает модель дополнительными сущностями.

## Бизнес-ограничения и правила качества данных

1. **`email` обязателен** при создании лида; формат проверяется приложением (не только тип `TEXT` в БД).
2. **`phone` обязателен** для лида и клиента; перед дедупликацией применяется нормализация.
3. **`source`** ограничен набором значений из столбца **ENUM** для поля `source` (в DDL — тип `lead_source`).
4. **`status`** ограничен набором значений из столбца **ENUM** для поля `status` (в DDL — тип `lead_status`).
5. Если **`status = 'Отложен'`**, поле **`follow_up_at` обязательно** (не `NULL`).
6. Если **`status` ∈ {'Ложный', 'Отказ'}**, **`close_reason` обязателен**.
7. При **`status = 'Успешно'`** должен существовать связанный **`Client`** для данного `Lead`.

## Дедупликация и повторные обращения (прикладной уровень)

- Проверка выполняется по **совпадению нормализованных `phone` или нормализованного `email`**.
- **Активные** для блокировки новой записи: статусы **`'Новый'`, `'В работе'`, `'Отложен'`**. При совпадении хотя бы по одному полю — **новый лид не создаётся**.
- Если совпадения **только** с лидами в **`'Ложный'`** или **`'Отказ'`** (активных нет), создаётся новый лид, заполняется **`reopen_of_lead_id`** (правило выбора «родителя» при нескольких закрытых — на реализации).
- Если **`phone`** указывает на один **активный** лид, а **`email`** — на **другой активный** лид, фиксируется **конфликт идентичности** до ручного разрешения.

Уникальность `phone` и `email` **глобально** одним `UNIQUE` на таблицу не выражает статусную логику; инварианты по «активным» дублям обеспечиваются приложением.

## Сущности и поля вне области модели

Намеренно отсутствуют как отдельные таблицы: отдельная сущность **задач** (follow-up ведётся в `Lead.follow_up_at`), журнал аудита статусов, расширенная RBAC, сделки, компании и прочие элементы CRM вне утверждённого scope.
