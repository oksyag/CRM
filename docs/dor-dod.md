# CRM для обработки лидов — Definition of Ready / Definition of Done

## О документе
Документ фиксирует прикладные DoR и DoD для user story в проекте CRM обработки лидов. Используется уже согласованный контекст: роли, статусы, поля, дедупликация, бизнес-правила и артефакты проекта (`overview.md`, `data-model.md`, `glossary.md`, `nfr.md`, `integration.md`, US-01..US-04, UC-01..UC-03). Общая agile-теория здесь не дублируется.

## Definition of Ready (DoR) для user story
User story считается готовой к взятию в работу, если выполнены все условия:

1. Заполнены ID, название, actor, trigger, предусловия, postconditions, основной сценарий, edge cases и критерии приёмки.
2. Поля, статусы, значения `source` и термины из истории совпадают с `overview.md`, `data-model.md` и `glossary.md` (без переименований и новых значений).
3. Используемые ошибки берутся из согласованного набора проекта: `400`, `409 duplicate`, `409 conflict`, опционально `429`, `500`, и `404` — там, где применимо.
4. Указаны связанные user stories и use cases (если связь есть).
5. Для интеграционных историй контракт согласован с `integration.md`; для историй про смену статусов и конвертацию — с разделением: `PATCH /leads/{id}/status` для `Отложен`/`Ложный`/`Отказ` и переходов с/в `Отложен`, `POST /leads/{id}/convert` — единственный путь в финальный статус `Успешно` с созданием `Client`.
6. История остаётся в рамках scope `overview.md`: новые сущности, статусы, роли, источники и эндпоинты в неё не вносятся без отдельного согласования.
7. Размер истории позволяет завершить её в одной итерации без скрытых зависимостей.

## Definition of Done (DoD) для user story
User story считается завершённой, если выполнены все условия:

1. Все критерии приёмки из истории выполнены и проверены.
2. Поведение соответствует жизненному циклу лида и согласованным переходам статусов; запрещённые переходы из финальных статусов (`Успешно`, `Ложный`, `Отказ`) реально запрещены.
3. Дедупликация работает по правилам проекта: совпадение по нормализованным `phone` или `email`, блокировка только при активных статусах (`Новый`, `В работе`, `Отложен`), повтор после `Ложный`/`Отказ` через `reopen_of_lead_id`, конфликт идентичности возвращает `409 conflict`.
4. Валидации соответствуют `nfr.md` и `integration.md`: обязательность `full_name`, `phone`, `email`, `source`; `consent=true` для веб-формы; формат `email`; нормализация `phone` и `email`; обязательность `follow_up_at` для `Отложен`; обязательность `close_reason` для `Ложный`/`Отказ`.
5. API использует только согласованные эндпоинты и коды ответа; новых эндпоинтов и статусов не появилось.
6. Логирование соответствует `nfr.md`: ключевые события зафиксированы, в записях есть `request_id`, полные значения `phone` и `email` в логах ошибок не сохраняются.
7. Документация (`overview`, `data-model`, `glossary`, `nfr`, `integration`, `functional/README`, US, UC) согласована с реализацией; расхождений между артефактами нет.
8. Поведение проверено на happy path и минимум на одном ключевом отказном сценарии из истории.
9. Регрессий по критериям приёмки соседних user stories не обнаружено.

## Пример применения: US-01 «Создание лида из веб-формы с валидацией и дедупликацией по телефону и email»

### DoR (US-01)
- Поля payload зафиксированы: `full_name`, `phone`, `email`, `consent=true`; `source=web` устанавливает CRM (`integration.md`).
- Активные статусы блокировки и правила повтора/конфликта определены в `overview.md` и `data-model.md`.
- Коды ответа описаны в US-01: `201`, `400`, `409 duplicate`, `409 conflict`, опционально `429`, `500`.
- Критерии приёмки сформулированы в `US-01-create-lead-from-web-form.md`.
- Сущность `Lead` и тип `lead_source` со значением `web` присутствуют в `data-model.md`.
- Контракт `POST /web/leads` описан в `integration.md`.

### DoD (US-01)
- `POST /web/leads` соответствует контракту из `integration.md`.
- `source=web` выставляется CRM; клиентское значение игнорируется.
- `phone` нормализуется; `email` валидируется по формату и нормализуется.
- Дедупликация работает: активный дубликат → `409 duplicate` с `existing_lead_id`; совпадение только с `Ложный`/`Отказ` → новый лид со статусом `Новый` и заполненным `reopen_of_lead_id`; разные активные лиды по `phone` и `email` → `409 conflict`.
- Невалидный payload или `consent != true` → `400`.
- При создании лида фиксируются `created_at` и `updated_at`.
- В логах присутствует `request_id`; полные `phone` и `email` в логах ошибок не сохраняются.
- Вручную проверены: happy path (создание нового лида), `400` (отсутствие обязательного поля или невалидный `email`), `409 duplicate` (активный дубликат).
