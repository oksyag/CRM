# UC-01 Регистрация нового лида

## ID
UC-01

## Цель
Зарегистрировать новый лид в CRM с проверкой обязательных полей, валидации `email` и дедупликации по нормализованным `phone` и `email`.

## Primary actor
В зависимости от канала создания: **Система** (прием заявки из веб-формы) или **Менеджер** (ручное создание в CRM).

## Secondary actors/systems
- **Веб-посетитель** — заполняет публичную форму (в сценарии веб-канала).
- **CRM API / хранилище** — прием запроса, валидация, дедупликация, сохранение лида.

## Trigger
- Получен запрос на создание лида из веб-формы (интеграция), **или**
- Менеджер выполнил действие «Создать лид» и отправил данные в CRM.

## Preconditions
- Интеграция веб-формы с CRM доступна (для веб-канала).
- Менеджер имеет доступ к функции создания лида (для ручного канала).
- Для всех сценариев создания указаны обязательные поля: `full_name`, `phone`, `email`, `source`.
- Для веб-формы дополнительно: `consent=true`, `source=web`.
- Для ручного создания поле `consent` не используется; `source` допустим только из списка `phone`, `chat`, `social`, `manual`. Значение `web` недопустимо.

## Postconditions
- **Успех:** создан лид со статусом `Новый` при отсутствии блокирующих дубликатов.
- **Дубликат активного лида:** новый лид не создан, возвращены сведения для сопоставления (`existing_lead_id`).
- **Повтор после закрытых лидов:** создан новый лид с заполнением `reopen_of_lead_id` (при выполнении условий ниже).
- **Конфликт идентичности:** новый лид не создан, зафиксирован конфликт для ручной проверки менеджером.

## Основной поток
1. Инициатор передает данные создания лида в CRM.
2. CRM проверяет обязательные поля и формат `email`.
3. CRM нормализует `phone` и `email` для дедупликации.
4. CRM определяет канал:
   - если с веб-формы — проверяет `consent=true` и `source=web`;
   - если вручную — проверяет, что `source` из `phone`, `chat`, `social`, `manual`, и что `source` не равен `web`.
5. CRM ищет **активный** лид с тем же `phone` **или** тем же `email` (активные статусы: `Новый`, `В работе`, `Отложен`).
6. Если активных совпадений нет — CRM создает лид со статусом `Новый`, фиксирует `created_at` и `updated_at`.
7. CRM возвращает идентификатор созданного лида и текущий статус.

## Альтернативные потоки
**A1 — Активный дубликат по `phone` или `email`**  
На шаге 5 найден активный лид по одному из полей. Новый лид не создается; возвращается ответ типа «дубликат» с указанием `existing_lead_id`.

**A2 — Повторное обращение после закрытия**  
На шаге 5 активных совпадений нет, но есть совпадение по `phone` и/или `email` только с лидами в статусах `Ложный` или `Отказ`. Создается новый лид со статусом `Новый`, заполняется `reopen_of_lead_id`.

**A3 — Конфликт двух активных записей**  
На шаге 5 один и тот же запрос сопоставляется так, что по `phone` найден один активный лид, а по `email` — другой активный лид. Новый лид не создается; возвращается конфликт для ручной проверки.

## Исключения
- Нарушены обязательность полей, формат `email` или требование `consent` для веб-канала (`400`).
- Для ручного канала указан недопустимый `source` (в том числе `web`) или неизвестное значение (`400`).
- Обнаружен активный дубликат по `phone` или `email` (`409 duplicate`).
- Конфликт идентичности между двумя активными лидами (`409 conflict`).
- Слишком частые попытки подачи с одних и тех же данных (защита от спама) — возможен ответ `429` (если включено на уровне реализации).
- Внутренняя ошибка CRM при сохранении (`500`).
- Отсутствие лида при последующих операциях вне этого сценария (`404`) — только если вызывается последующее чтение по несуществующему идентификатору.

## Связанные user stories
- **US-01** — создание лида из веб-формы с валидацией и дедупликацией по телефону и email.
- **US-02** — создание лида вручную менеджером (каналы без `web`).

## Связанные данные / API / бизнес-правила
**Данные**
- Сущность `Lead`: обязательные `full_name`, `phone`, `email`, `source`; начальный `status=Новый`; при повторе после закрытых — `reopen_of_lead_id`.
- Статусы лида: `Новый`, `В работе`, `Отложен`, `Успешно`, `Ложный`, `Отказ`.

**API (минимально, по следам traceability проекта)**
- `POST /web/leads` — создание из веб-формы.
- `POST /leads` — ручное создание менеджером.
- `GET /leads/{id}` — получение карточки после создания (при необходимости).

**Бизнес-правила**
- Дедупликация: совпадение по нормализованному `phone` **или** `email`; блокировка создания только при активных статусах `Новый`, `В работе`, `Отложен`.
- После финальных статусов `Ложный`/`Отказ` возможен новый лид и связка через `reopen_of_lead_id`.
- Развести каналы: `web` только для интеграции формы; ручной ввод — только `phone` | `chat` | `social` | `manual`.
