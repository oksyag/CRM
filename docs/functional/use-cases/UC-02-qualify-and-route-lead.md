# UC-02 Активная обработка лида (квалификация и маршрутизация)

## ID
UC-02

## Цель
Закрепить лид за ответственным менеджером и перевести его из статуса `Новый` в `В работе`, чтобы начать квалификацию и дальнейшую обработку по согласованному жизненному циклу.

## Primary actor
**Менеджер по продажам.**

## Secondary actors/systems
- **CRM** — валидация перехода статуса, проверка менеджера, сохранение изменений.
- **Система** — не инициирует этот сценарий; участвует как исполнитель бизнес-правил на стороне CRM.

## Trigger
Менеджер в карточке лида (или через эквивалентное действие в UI) назначает ответственного и подтверждает перевод лида в работу.

## Preconditions
- Лид существует в системе.
- Текущий статус лида — `Новый` (как в **US-03**).
- Выбранный для назначения менеджер существует и активен (`is_active`).
- Финализация (`Отложен`, `Успешно`, `Ложный`, `Отказ`) в этом use case **не выполняется** — она относится к **UC-03** и **US-04**.

## Postconditions
- У лида заполнено поле ответственного (`assignee_id`).
- Статус лида — `В работе`.
- Обновлено поле `updated_at` у лида.

## Основной поток
1. Менеджер находит лид (в т.ч. через список с фильтрами по `status`, менеджеру, `source` — в рамках уже согласованного scope).
2. Менеджер открывает карточку лида со статусом `Новый`.
3. Менеджер выбирает ответственного менеджера (`assignee_id`).
4. CRM проверяет, что выбранный менеджер существует и активен.
5. CRM сохраняет назначение и выполняет переход статуса `Новый` → `В работе`.
6. CRM отображает обновленную карточку: ответственный и статус `В работе`.

## Альтернативные потоки
**A1 — Повторное выполнение при уже выполненном назначении**  
Если лид уже в `В работе` у того же ответственного, операция считается идемпотентной с точки зрения бизнес-результата (нет повторной смены смысла статуса).

## Исключения
- Лид не найден (`404`).
- Некорректный или неактивный `assignee_id` (`400`).
- Перевод в `В работе` запрещен, если лид в финальном статусе (`Успешно`, `Ложный`, `Отказ`) — `409`.
- UC-02 описывает переход только из `Новый`. Возврат `Отложен` → `В работе` относится к UC-03 (поток A1) и здесь не покрывается. Из финальных статусов (`Успешно`, `Ложный`, `Отказ`) перевод в `В работе` запрещён и возвращает `409` (`transition_forbidden`).

## Связанные user stories
- **US-03** — назначение ответственного и перевод `Новый` → `В работе`.

## Связанные данные / API / бизнес-правила
**Данные**
- `Lead.assignee_id`, `Lead.status`, `Lead.updated_at`.
- Активные статусы жизненного цикла: `Новый`, `В работе`, `Отложен` (другие переходы — в других UC).

**API (по следу traceability проекта)**
- `GET /managers` — список активных менеджеров для выбора ответственного.
- `PATCH /leads/{id}/assignee` — назначение ответственного.
- `PATCH /leads/{id}/status` — смена статуса на `В работе` (после успешной валидации исходного статуса `Новый`).
- `GET /leads?status=&manager=&source=` — поиск лида перед открытием карточки.

**Бизнес-правила**
- Назначение и старт активной обработки оформляются как единый управляемый шаг из **US-03**.
- Дедупликация и регистрация лида не входят в UC-02 (см. **UC-01** / **US-01**, **US-02**).
