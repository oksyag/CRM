# US-02 Создание лида вручную менеджером

## ID
US-02

## Название
Создание лида вручную менеджером.

## User story
Как менеджер по продажам,  
я хочу завести лид в CRM вручную (звонок, чат, социальные сети, личное обращение),  
чтобы заявка не терялась и в активной работе не появлялись дубли.

## Бизнес-ценность
- Захват лидов из каналов, где нет веб-формы.
- Единый процесс качества данных и дедупликации как для входа из сайта.

## Actor
Менеджер.

## Trigger
Менеджер отправляет форму создания лида через интерфейс CRM или соответствующий API-запрос.

## Предусловия
- Менеджер авторизован в системе (детали аутентификации вне объёма этого проекта).
- Переданы обязательные поля: `full_name`, `phone`, `email`, `source`.
- Поле `source` принимает одно из допустимых для ручного создания значений: `phone`, `chat`, `social`, `manual`. Значение `web` недопустимо (создание из сайта — US-01).
- Формат `email` должен быть валидным.
- Для дедупликации используются нормализованные значения `phone` и `email`.
- При создании вручную поле согласия с веб-формы (`consent`) не используется.

## Postconditions
- Если активный дубликат не найден, создан новый лид со статусом `Новый` и указанным `source`.
- Если найден активный дубликат (`Новый`, `В работе`, `Отложен`) по `phone` или `email`, новый лид не создаётся; возвращается признак дубликата и `existing_lead_id`.
- Если совпадения есть только по лидам со статусом `Ложный` или `Отказ`, создаётся новый лид с заполнением `reopen_of_lead_id`.
- Если `phone` и `email` указывают на разные активные лиды, возвращается конфликт для ручной проверки.

## Основной сценарий
1. Менеджер вводит данные лида (`full_name`, `phone`, `email`, источник/канал) и выбирает `source`:
   - звонок -> `phone`;
   - чат -> `chat`;
   - социальные сети -> `social`;
   - личное обращение/офлайн -> `manual`.
2. CRM валидирует обязательные поля и формат `email`.
3. CRM нормализует `phone` и `email` для проверки.
4. CRM проверяет наличие активного лида с тем же `phone` или `email`.
5. Если активного дубликата нет, CRM создаёт лид со статусом `Новый`, фиксирует `created_at` и `updated_at`.
6. CRM показывает подтверждение успешного создания с идентификатором лида и статусом `Новый`.

## Edge cases / ограничения
- При указании `source=web` или недопустимого значения `source` операция отклоняется (`400`).
- При отсутствии обязательного поля или невалидном `email` операция отклоняется (`400`).
- При активном совпадении по `phone` или `email` возвращается `409 duplicate`.
- Если `phone` и `email` совпали с разными активными лидами, возвращается `409 conflict`.
- При внутренней ошибке сохранения возвращается `500`.

## Критерии приемки
- При корректно заполненных обязательных полях, `source` из набора `phone` / `chat` / `social` / `manual` и отсутствии активного дубликата создан лид со статусом `Новый` и выбранным `source`.
- При попытке сохранить лид с `source=web` создание не выполняется, отображается ошибка (аналог ответа `400`).
- При отсутствии или неверном формате `email` лид не создаётся, пользователю показывается ошибка валидации (аналог ответа `400`).
- При совпадении `phone` или `email` с лидом в статусах `Новый`/`В работе`/`Отложен` новый лид не создаётся, возвращается `409` и `existing_lead_id`.
- При совпадении `phone` или `email` только с лидами в статусах `Ложный`/`Отказ` создаётся новый лид с заполненным `reopen_of_lead_id`.
- При конфликте идентичности (`phone` и `email` указывают на разные активные лиды) возвращается `409 conflict` и требуется решение менеджера вручную.
- После успешного создания менеджер видит карточку или подтверждение с `lead_id` и статусом `Новый`.
