# CRM для обработки лидов — интеграция веб-формы

## О документе
Документ описывает интеграцию публичной веб-формы сайта с CRM: источник данных, контур обмена, контракт запроса, обязательные поля, базовые валидации, дедупликацию, обработку ошибок и поведение при повторной отправке. Поля и термины согласованы с `overview.md`, `data-model.md`, `glossary.md` и `nfr.md`. Полная техническая спецификация API выносится в `api/openapi.yaml`.

## Источник данных и контур интеграции
- **Источник:** публичная веб-форма на сайте, заполняемая веб-посетителем.
- **Получатель:** CRM API, эндпоинт `POST /web/leads`.
- **Канал:** одиночный синхронный HTTP-вызов «веб-форма → CRM API» по HTTPS.
- **Промежуточные системы:** не используются. Очередей, шин событий и сторонних сервисов в этом контуре нет.

## Контракт запроса
- **Метод и URL:** `POST /web/leads`.
- **Транспорт:** только HTTPS (см. `nfr.md`, раздел «Безопасность»).
- **Content-Type:** `application/json; charset=utf-8`.
- **Аутентификация веб-формы перед CRM:** определяется на уровне реализации (типичный вариант — серверный API-ключ для формы). Security-схема в OpenAPI намеренно не описывается; контур доступа задаётся конфигурацией среды.
- **Источник лида:** для этого канала CRM фиксирует `source = web` сама; клиентская сторона значение `source` не задаёт.

## Обязательные поля формы и payload
Поля payload соответствуют атрибутам сущности `Lead` из `data-model.md`. Поле `source` для веб-канала всегда `web` и устанавливается CRM, а не клиентом формы.

| Поле payload | Поле `Lead` | Обязательность | Комментарий |
|---|---|---|---|
| `full_name` | `Lead.full_name` | да | ФИО или имя контакта. |
| `phone` | `Lead.phone` | да | Нормализуется CRM перед дедупликацией. |
| `email` | `Lead.email` | да | Формат проверяется CRM; нормализуется перед дедупликацией. |
| `consent` | бизнес-правило веб-канала | да | Должно быть `true`. |
| `source` | `Lead.source` | задаётся CRM | Для веб-канала всегда `web`. Клиент не передаёт это значение, либо оно игнорируется CRM. |

Пример минимального payload:

```json
{
  "full_name": "Иван Иванов",
  "phone": "+7 (901) 123-45-67",
  "email": "ivan@example.com",
  "consent": true
}
```

## Базовые валидации
1. Все обязательные поля присутствуют и непустые: `full_name`, `phone`, `email`, `consent`.
2. `consent == true`. Значение `false` или отсутствие — отказ в создании.
3. `email` соответствует базовому формату email; валидация выполняется на стороне CRM независимо от валидации в браузере (см. `nfr.md`, раздел «Безопасность»).
4. `phone` приводится к нормализованной форме перед дедупликацией.
5. `email` нормализуется перед дедупликацией: минимально — trim и единый регистр.
6. `source` устанавливается CRM как `web`; клиентское значение игнорируется.

## Дедупликация
Правила полностью совпадают с `overview.md` и `data-model.md`:

- Проверка выполняется по совпадению **нормализованного `phone`** или **нормализованного `email`**.
- **Активные статусы**, при которых новый лид **не создаётся**: `Новый`, `В работе`, `Отложен`. Возвращается `409 duplicate` и идентификатор уже существующего лида (`existing_lead_id`).
- Если совпадения есть **только** с лидами в статусах `Ложный` или `Отказ`, создаётся **новый лид** со статусом `Новый` и заполненным `reopen_of_lead_id` (повторное обращение).
- Если `phone` указывает на один **активный** лид, а `email` — на **другой активный** лид, фиксируется **конфликт идентичности**: новый лид не создаётся, требуется ручная проверка менеджером (`409 conflict`).

## Обработка ошибок
| Код | Когда возвращается | Что показывает веб-форма пользователю |
|---|---|---|
| `201` | Лид создан, в том числе как повторное обращение с заполненным `reopen_of_lead_id`. | Подтверждение успешной отправки заявки. |
| `400` | Не хватает обязательного поля, неверный формат `email`, `consent != true`. | Сообщение валидации с указанием, что исправить. |
| `409 duplicate` | Найден активный дубликат по `phone` или `email`. | Сообщение, что заявка уже зарегистрирована; передаваемый `existing_lead_id` в UI пользователю не показывается. |
| `409 conflict` | Конфликт идентичности (разные активные лиды по `phone` и `email`). | Нейтральное сообщение, что заявка получена и будет обработана менеджером вручную. |
| `429` | Опциональный anti-spam guard при частых повторах с одних данных. | Сообщение «попробуйте позже». |
| `500` | Внутренняя ошибка CRM при сохранении. | Общее сообщение об ошибке с предложением повторить попытку позже. |

Сообщения веб-формы для пользователя формулируются простым языком и не раскрывают внутренние идентификаторы и детали системы. Полные значения `phone` и `email` в логах ошибок не сохраняются (см. `nfr.md`, разделы «Безопасность» и «Логирование»).

## Поведение при повторной отправке
- **Повтор того же payload при активном лиде:** CRM возвращает `409 duplicate` с тем же `existing_lead_id`; новый лид не создаётся.
- **Повтор после ранее закрытого лида (`Ложный`/`Отказ`):** создаётся новый лид со статусом `Новый` и заполненным `reopen_of_lead_id` — это валидное повторное обращение, а не дубликат.
- **Защита от спама:** опциональный anti-spam guard может вернуть `429`, если CRM фиксирует частые повторы с одинаковыми `phone`/`email` за короткий интервал. Повторная отправка после паузы допустима.
- **Идемпотентность:** для `POST /web/leads` поддерживается заголовок `Idempotency-Key`. При повторе запроса с тем же ключом CRM возвращает результат предыдущей обработки и не создаёт новый лид. Дополнительно корректность повторов обеспечивается правилами дедупликации.

## Минимальный flow интеграции
1. Веб-посетитель заполняет форму на сайте, отмечает согласие на обработку данных и отправляет форму.
2. Сайт отправляет `POST /web/leads` по HTTPS с минимальным payload (`full_name`, `phone`, `email`, `consent=true`).
3. CRM проверяет обязательные поля, формат `email` и `consent=true`.
4. CRM нормализует `phone` и `email` и выполняет дедупликацию.
5. CRM:
   - создаёт новый лид со статусом `Новый` и `source=web`, либо
   - создаёт новый лид со статусом `Новый` и заполненным `reopen_of_lead_id` (повторное обращение), либо
   - возвращает `409 duplicate` с `existing_lead_id`, либо
   - возвращает `409 conflict` (конфликт идентичности).
6. CRM фиксирует `created_at` и `updated_at` у созданного лида.
7. Веб-форма получает ответ и показывает пользователю сообщение успеха или ошибки.

## Логирование интеграции
- Логируются ключевые события вызова `POST /web/leads`: создание лида, повторное обращение (`reopen_of_lead_id`), отказ по дубликату, конфликт идентичности, технические ошибки.
- В лог попадают: `timestamp`, код ответа, тип события, `request_id`, признак `source=web`.
- Полные значения `phone` и `email` в логах ошибок не сохраняются.
