# CRM для обработки лидов — Нефункциональные требования (NFR)

## О документе
Документ фиксирует базовые нефункциональные требования для CRM обработки лидов в рамках согласованного scope.

## Производительность
- Время ответа API для типовых операций (`create lead`, `get lead`, `update status`) — до 1 секунды при нормальной нагрузке.
- Время ответа списка лидов с фильтрами по статусу, менеджеру и источнику — до 2 секунд при объеме до 10 000 лидов.
- Проверка дедупликации по `phone` и `email` выполняется в рамках одного запроса создания лида.

## Доступность
- Целевой уровень доступности: не ниже 95% в месяц.
- При временной недоступности API пользователь веб-формы получает понятное сообщение об ошибке и может повторить отправку.

## Безопасность
- Передача данных между веб-формой и CRM выполняется только по HTTPS.
- Сервер выполняет обязательную валидацию входных данных, независимо от проверок на клиенте.
- Хранятся только необходимые персональные данные: `full_name`, `phone`, `email`.
- В логах ошибок не сохраняются полные значения `phone` и `email`.

## Логирование
- Логируются ключевые события: создание лида, отказ по дубликату, смена статуса, конвертация, закрытие.
- Для ошибок фиксируются: `timestamp`, код ошибки, тип ошибки, `request_id`.
- Логи не должны содержать полные персональные данные.

## Качество данных
- При создании лида обязательны поля: `full_name`, `phone`, `email`, `source`.
- Для лида из веб-формы дополнительно обязательно `consent=true`.
- Поле `phone` нормализуется перед дедупликацией.
- Поле `email` валидируется по формату и участвует в дедупликации.
- Значение `status` допускается только из согласованного набора: `Новый`, `В работе`, `Отложен`, `Успешно`, `Ложный`, `Отказ`.
- Для статуса `Отложен` обязательно заполнение `follow_up_at`.
- При финальных статусах `Ложный` и `Отказ` причина закрытия хранится как свободный текст.
- Повторное обращение после `Ложный`/`Отказ` создается как новый лид с заполнением `reopen_of_lead_id`.
- Если `phone` и `email` совпали с разными активными лидами, запрос помечается как конфликт и требует ручной проверки.

## Usability
- Веб-форма создания лида содержит только необходимые поля и понятные подписи.
- Сообщения валидации и ошибок формулируются простым языком и указывают, что исправить.
- Список лидов поддерживает фильтры по `status`, `manager`, `source`.
- В карточке лида явно отображаются: текущий статус, ответственный менеджер, `follow_up_at` (если задан).

## Поддерживаемость
- Термины и поля (`Lead`, `Client`, `status`, `reopen_of_lead_id`) используются единообразно во всех артефактах.
- Изменения бизнес-правил синхронно отражаются в документации, API-спецификации, SQL и диаграммах.
- Документация хранится в Markdown и поддерживается как doc as code.

## Ограничения и допущения
- Проект ограничен согласованным scope: без enterprise-функций и без расширенного интеграционного контура.
- Роли в системе ограничены `Менеджер` и `Система`.
- Дедупликация выполняется по `phone` и `email`.
- Follow-up хранится как часть сущности лида (`follow_up_at`), без отдельной сущности задач.
- Интеграция ограничена контуром веб-форма -> CRM API.
